# Module: Server (`server/`)

## Purpose
Central hub. Receives jobs via REST+WS, queues them in SQLite, spawns AI CLI tools as subprocesses, streams results back to bridges+clients. Manages all state.

## Entry Point & Startup (`src/index.ts`)
1. Load config from env vars
2. Open SQLite DB + run migrations
3. Initialize all 14 repos (jobs, agents, apikeys, users, policies, audit, projects, workers, usage, dependencies, headlessPrograms, settings, sync, apiBridges)
4. Seed defaults on first run (bootstrap admin user with generated/random password saved to `bootstrap-admin.txt` beside DB unless `BOOTSTRAP_ADMIN_PASSWORD` is provided and strong, plus admin API key). Agent configs now start empty by default on first install. Example headless programs can be seeded (`SEED_EXAMPLE_HEADLESS_PROGRAMS=true`, default on). Always seeds coordinator scripts dir (`data/coordinator-scripts/`) and coordinator playbooks dir (`data/coordinator-playbooks/`) with missing defaults on startup (global + blender + godot + houdini + unity + unreal + comfyui starter playbooks). Pulls API bridge skills at startup: loops through `apiBridgesRepo.listEnabled()` and calls `pullBridgeSkills()` for each enabled API bridge program.
4.5 Normalize legacy agent args at startup - Codex (`-p`, old approval flags) and Claude Code (`-p`, `--print`, `--model`) - so existing DB configs remain compatible with current CLI behavior.
5. Create WebSocket hub, ProcessTracker, Scheduler, SyncManager
6. Start timeout checker, sync cleanup timer, session cleanup timer (hourly), worker loop, and coordinator training scheduler tick (60s)
7. Create Hono app, mount all routes
8. `Bun.serve()` handles HTTP (→Hono) and WS (at `/ws`) - **Optional TLS** via `TLS_CERT_PATH` + `TLS_KEY_PATH` env vars. **WS auth enforced before upgrade** (API key validated in fetch handler; supports query `key` and `Sec-WebSocket-Protocol: arkestrator.auth.<token>`). WS type/role constraints are enforced (`client` sockets reject `bridge` keys; `bridge` sockets reject `client` keys). Runs with a single listener (no `reusePort`) to avoid split traffic across stale processes. On bridge connect: broadcasts bridge status + upserts worker + **upserts worker_bridges** (program history). Worker identity uses the socket-supplied `workerName` when present, with local fallback derivation (`workerName` → `osUser@ip` → `host-ip` → name/program) only when missing; the server no longer merges/deletes workers solely because sockets share an IP or OS username. On client connect: sends full bridge context sync via `hub.sendContextSync()` + **`hub.sendBridgeStatus()`** + **`hub.sendWorkerStatus()`** (so client gets full state immediately). On bridge disconnect: hub clears context + broadcasts `bridge_context_clear` to clients. On client disconnect: calls `handleClientDisconnect()` from `client-dispatch.ts` to fail any orphaned dispatched local LLM jobs.
9. Graceful shutdown on SIGINT/SIGTERM (clears all timers)

### Job Queue SQL (pickNext)
```sql
SELECT * FROM jobs WHERE status = 'queued'
  AND id NOT IN (
    SELECT d.job_id FROM job_dependencies d
    JOIN jobs j ON j.id = d.depends_on_job_id
    WHERE j.status NOT IN ('completed')
  )
  ORDER BY CASE priority WHEN 'critical' THEN 0 WHEN 'high' THEN 1 WHEN 'normal' THEN 2 WHEN 'low' THEN 3 END, created_at ASC
  LIMIT 1
```

## Skills System (`src/skills/`, `src/db/skills.repo.ts`, `src/routes/skills.ts`)

Skills are modular instruction snippets that customize how the coordinator agent works with bridges. They are injected into the agent's system prompt at job execution time.

### Skill Categories

| Category | Purpose | Examples |
|---|---|---|
| `coordinator` | Core orchestration scripts — the "brain" for each bridge connection. Define the transport gate, execution loop, quality checks, and verification requirements. | Global Coordinator, Blender Coordinator |
| `bridge` | Bridge-specific operational knowledge — API patterns, best practices, quirks for a DCC app. Pulled from bridge repos. | Blender materials & shading, Houdini SOP networks, Godot scene management |
| `training` | Auto-generated from training runs. When users rate completed jobs and run training analysis, learned patterns are extracted and stored as skills. | "Always deselect all before selecting in Blender", "Use Lock/Unlock around Fusion comp edits" |
| `playbook` | Step-by-step workflow recipes for common multi-step tasks. Think "how-to" guides the agent follows. | "Set up a character rig in Blender", "Create a particle system in Houdini" |
| `verification` | Quality checks and validation rules applied before marking a job complete. | "Verify render output exists and is > 0 bytes", "Check scene has at least one light source" |
| `project` | Project-specific context — conventions, file structure, naming rules tied to a particular project. | "This project uses PBR materials only", "All models go in `assets/models/`" |
| `custom` | User-created skills that don't fit the above categories. Catch-all for anything custom. | Any user-defined instruction snippet |

Registers/deregisters a virtual bridge in the WebSocketHub so ComfyUI appears in the bridge status list when running. Started automatically on server boot, stopped on graceful shutdown.
- **utils/spawn.ts**: `spawnWithFallback()` retries Windows command shims/extensions (`.cmd`, `.exe`, `.bat`) when plain command spawn returns `ENOENT` (e.g. `codex` on Windows PATH shims). `SpawnWithFallbackOptions` type defined explicitly (avoids `Parameters<typeof Bun.spawn>` issue).
- **process-tracker.ts**: Tracks running subprocesses, 30s timeout check interval, kill on timeout/shutdown. Constructor accepts `number | (() => number)` for dynamic timeout (supports runtime-configurable job timeout). `getProcess(jobId)` exposes the subprocess handle for a given job (used for stdin guidance delivery). `suspend(jobId)` and `resume(jobId)` methods allow temporarily removing a job from concurrency accounting (suspended jobs don't count toward slots), enabling training parent jobs to free a slot while polling child analysis.
- **file-snapshot.ts**: Recursive directory snapshot (skips .hidden, node_modules, __pycache__) with robust post-run diffing: reconciles fs.watch events with before/after snapshots, reports binary changes as metadata placeholders (`[binary:ext] N.NMB` with file extension detection), and tracks symlink changes.
- **stream-json-parser.ts**: Parses Claude Code `--output-format stream-json --verbose` and Codex CLI `--json` JSONL output. Extracts human-readable display lines for real-time log streaming. Text preview limit: 500 chars. Tool result preview: 500 chars. Formats MCP tool calls (`arkestrator__execute_command`, `arkestrator__create_job`, etc.) with meaningful summaries. Token extraction accumulates per-turn usage from assistant events, extracts from `usage` sub-object in result events, and supports `total_cost`/`cost_usd` aliases for cost reporting. Codex JSONL events handled: `thread.started`, `turn.started`, `item.completed` (accumulates token usage from `usage` fields), `turn.completed`.
- **token-parser.ts**: Regex-based token extraction from stdout (claude-code, codex, gemini patterns)

## Queue System (`src/queue/`)
- **worker.ts**: `WorkerLoop` - adaptive `setTimeout` poll loop (base/min/max cadence). Speeds up when busy or dispatching, backs off when idle. `WorkerDeps` now includes optional `skillIndex` field and `policiesRepo`, both passed through to spawner. Each tick: check available slots (maxConcurrent - running), pick next job, claim atomically, **check per-user token budget** (if job has `submittedBy` and user has limits, fails job if over budget), load agent config + policies, apply per-job runtime overrides to produce effective spawn config, inject worker's projectRoot if targeting, and dispatch. `dispatchJob()` routes task-mode jobs (`job.mode === "task"`) to `TaskExecutor` instead of `spawnAgent()`, enabling non-agentic task execution without spawning an AI subprocess. The tick loop now also collects connected bridge worker names and uses `pickNextForWorker` per worker to ensure worker-targeted jobs only dispatch when the target worker has bridges connected. After the dispatch loop, the worker checks if queued jobs still exist and keeps the poll interval aggressive (`minPollMs`) instead of backing off, preventing dispatch starvation where worker-targeted jobs stay queued for minutes. The loop now also passes the shared worker-resource lease manager into spawned runs so DCC-heavy bridge/headless steps can serialize per worker without globally capping agent concurrency. Tick body wrapped in try/catch for resilience. Periodic diagnostic logging (every 10s) shows slot usage + current poll interval. **`dispatchById(jobId)`** public method allows manual dispatch of a specific queued job, bypassing the poll cycle (used by `POST /api/jobs/:id/dispatch`).
- **scheduler.ts**: Thin wrapper around `jobsRepo.pickNext()` and `jobsRepo.pickNextForWorker(workerName)` for worker-targeted scheduling.
- **retry-policy.ts**: Job retry policy helpers. `isTransientError(error)` classifies transient failures (timeouts, bridge disconnects, spawn errors). `computeRetryDelay(retryCount)` returns exponential backoff delay. Default policy: `maxRetries=2` for regular jobs, 30min TTL for targeted jobs. Worker loop expires stale targeted jobs every 60s.

## Workspace Resolution (`src/workspace/`)
- **resolver.ts**: Simplified fallback chain: preferredMode → server default → no projectRoot → explicit projectId prompt injection → local path exists → attached files → command fallback. Projects now only attach for prompt injection (via `formatProjectPrompt()`) rather than workspace resolution; no bridge-path glob matching or project-based workspace mode override. Resolver returns `resolutionStep` + `resolutionReason` for diagnostics; spawner warns when step 5+ fallback branches are used. Jobs that set `editorContext.metadata.project_selection = "none"` skip project lookup + local-path repo resolution and fall through to sync/command branches.
- **command-mode.ts**: Builds command-mode system prompt (tells agent to output fenced code blocks), parses output for code fences, detects bridge type from metadata or `bridgeProgram` fallback (supports godot, blender, houdini, unity, comfyui), derives the expected executable script language, and filters out echoed helper/template snippets (`am`, curl, PowerShell, `# your code here`) before returning commands. ComfyUI command-mode instructions explicitly allow Python-driven filesystem operations for outputs/model installs (while still forbidding Bash/Write/Edit tool usage) and now clarify machine boundaries: destination paths are local to the ComfyUI worker unless artifacts are explicitly transferred to a destination worker bridge.
- **sync-manager.ts**: Creates temp dirs, writes attached files, TTL-based cleanup

## Policy Enforcement (`src/policies/enforcer.ts`)
- `checkPromptFilters()` - regex against prompt text
- `checkCommandScripts()` - regex against command script content (case-insensitive). Filters at 4 choke points: **real-time stream monitoring** (kills agent on Bash tool violation), spawner (command-mode output), REST bridge-command endpoint, WS bridge_command_send handler
- `checkEngineModel()` - exact match on engine or engine:model
- `checkFilePaths()` - glob matching via minimatch (normalized to forward slashes)
- `getToolRestrictions()` - returns blocked tool names
- `validateJobSubmission()` - master check combining prompt + engine/model

### Real-Time Stream Policy Enforcement
The stream-json-parser (`src/agents/stream-json-parser.ts`) accepts an optional `CommandPolicyChecker` callback that checks Bash commands against command_filter policies as they stream from the agent CLI. When a violation is detected:
1. The `ParsedLogLine` includes a `policyViolation` field
2. The spawner immediately kills the agent process (`proc.kill()`)
3. The job is failed with a policy violation error message
4. Job logs include `[POLICY VIOLATION]` entries for auditing

The MCP `execute_local` tool (`src/mcp/tool-server.ts`) also checks command_filter policies using the calling job's submitter for user-scoped policy support.

### Policy Scoping
Policies support three scopes: `global` (all users/projects), `user` (per-user), `project` (per-project). Context-aware loading via `PoliciesRepo.getEffectiveForContext(userId, projectId)` merges all applicable scopes.

### Resource Management Policies
Four resource policy types enforce runtime limits:
- `concurrent_limit` — max concurrent running jobs (pattern: number, e.g. `"3"`)
- `process_priority` — OS process priority for agent processes (pattern: `"low"` | `"below_normal"` | `"normal"` | `"above_normal"` | `"high"`)
- `token_budget` — max token usage (pattern: `"input:500000"` | `"output:100000"` | `"total:600000"`)
- `cost_budget` — max API cost in USD (pattern: `"5.00"`)

Concurrent limits are checked at dispatch time in worker.ts (job stays queued if over limit). Process priority applied best-effort after spawn via `applyProcessPriority()` in `src/utils/spawn.ts` (Windows: wmic, Unix: renice). Token/cost budgets checked at job submission.

Primary output: `client/src-tauri/binaries/arkestrator-server-{triple}[.exe]`.  The binary embeds the Bun runtime, SQLite, and all dependencies - no Bun or Node.js needed on the target machine. Used by the Tauri client's "Start Local Server" feature in production mode.

## Dev Reset Script
`pnpm --filter server reset:state` (or `bun scripts/reset-state.ts`) performs a local reset for stuck dev sessions:
- kills listeners on ports `7800` (server) and `5173` (client)
- removes `server/data` (SQLite DB + temp state)
- removes shared bridge config `~/.arkestrator/config.json`
- supports `--dry-run` to preview actions
- on Windows, locked files are retried and treated as best-effort (reset continues instead of failing hard)

## MCP Tool Server (`src/mcp/`)
Exposes Arkestrator bridge interaction and job orchestration as MCP tools for external MCP clients (for example Codex app) and coordinator agents. Mounted at `/mcp`.

- **tool-server.ts**: `createMcpServer()` - creates `McpServer` instance with 17 tools. `McpDeps` includes `jobsRepo`, `agentsRepo`, `depsRepo`, `apiBridgesRepo`, optional `processTracker`, `principal` (authenticated user or API key), and request-auth forwarding metadata (`requestOrigin`, `requestAuthHeader`, `requestCookieHeader`) used by client-parity forwarding. Uses `homedir` from `os` for default download directory resolution. Per-tool permission checks: `execute_command`/`execute_multiple_commands`/`run_headless_check` require `executeCommands`, `create_job`/`cancel_job` require `submitJobs`, `submit_job_intervention` requires `interveneJobs`.
  - *Bridge tools:* `execute_command`, `execute_multiple_commands`, `list_bridges`, `list_targets`, `get_bridge_context`
  - *Job orchestration tools:* `list_agent_configs`, `create_job`, `get_job_status`, `get_job_logs`, `list_jobs`, `cancel_job`
  - *Task job tools:* `create_task` (create a single non-agentic task job with execution type, command, args, program, target worker), `create_tasks` (batch create up to 20 task jobs). `resolveTaskRef()` helper resolves `#T<N>` short references to job IDs for task tracking.
  - *API bridge tools:* `execute_api_bridge` (execute an action on a configured API bridge by name; accepts optional `download_dir` parameter; auto-downloads output files in fire-and-forget mode; response includes `downloadedTo` directory, `localPath` per file, and `downloadStatus: "downloading"` instead of raw URLs/`downloadHint`), `list_api_bridges` (list all configured API bridges with their available actions)
  - *Client parity tool:* `client_api_request` (method/path/body) - forwards to an allowlisted set of non-admin client endpoints so MCP can reuse existing route validation/permissions for rich job payloads and training workflows.
  - *Running-job guidance tools:* `submit_job_intervention` and `list_job_interventions` forward to the same intervention APIs used by the client so MCP users can steer queued/paused jobs and supported running jobs with the same delivery-state semantics.
  - `create_job` creates a sub-agent job with optional `target_program`, `target_worker`, `depends_on_job_ids`, `coordination_scripts` (mode + selected script names for per-sub-job coordination script control), **`handover_notes`** (context from parent, prepended to prompt as `## Context from Coordinator`). Sub-jobs default to `coordinationMode: "server"`. Notifies clients via `hub.broadcastToType`.
  - `get_job_status` returns status, file changes, commands, blocking deps, **`output_summary`** (last 20 lines of logs for completed jobs, last 10 for failed).
  - `get_job_logs` returns tail logs for any job with bounded line limits (`lines` arg, max 2000), plus `total_lines`/`returned_lines`.
  - `list_jobs` returns recent jobs (up to 50) with optional status filter.
  - `cancel_job` cancels queued/paused/running jobs; for running jobs it kills tracked subprocesses first when process tracking is wired.
  - `list_targets` returns orchestration-ready targets as a union of currently connected bridges, enabled headless program configs, and configured API bridges.
  - `run_headless_check` now uses shared `runHeadlessCheck()` from `routes/bridge-commands.ts` so MCP and REST/CLI return consistent behavior/output.
  - All tools call shared helpers from `routes/bridge-commands.ts` or repos directly.

- **stateless-transport.ts**: Extracted `StatelessTransport` class — in-memory, Bun-compatible MCP transport (no Node.js `http.ServerResponse` dependency). Reused by both `routes.ts` (HTTP MCP endpoint) and `in-process-client.ts` (server-side local agentic loops).

- **in-process-client.ts**: In-process MCP client for server-side local agentic loops. Creates an `McpServer` + `StatelessTransport` pair connected in-memory, allowing the spawner to call MCP tools without HTTP round-trips. Exports: `createInProcessMcpClient()` (factory), `InProcessMcpClient` interface (with `listTools()` and `callTool()` methods).

- **routes.ts**: `createMcpRoutes()` - Hono route handler at `/mcp`. Imports `StatelessTransport` from `stateless-transport.ts`. Each POST creates a fresh transport, connects to MCP server, delivers JSON-RPC message, captures response. Auth uses `principalHasPermission("useMcp")` uniformly for both user sessions and API keys (was role-based for API keys); auto-provisioned user API keys (`auto:user:<id>`) are also gated by that same user permission. Passes `principal` into `McpDeps` for per-tool permission enforcement. **Bug fix**: MCP notifications (no `id` field, e.g. `notifications/initialized`) now return HTTP 202 immediately instead of hanging - `handleMessage()` checks for presence of `id` field and returns `null` for notifications.

### How agents discover MCP tools
The spawner writes `.mcp.json` to the job's CWD before spawning Claude Code agents. Claude Code auto-discovers MCP servers from this file. The config is merged with any existing `.mcp.json` and restored/removed after the job completes.

## HTTP File Transfers (`src/transfers/`)
- **transfer-manager.ts**: `TransferManager` — manages HTTP-based file transfers between server and clients. Tracks active transfers in-memory, handles temp file storage and cleanup, generates transfer IDs, and coordinates with the WS layer to send `transfer_initiate` messages to clients. Supports both upload (client→server) and download (server→client) flows. Configurable via `transferTempDir` and `maxTransferSizeMb` in config.
