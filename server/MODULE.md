# Module: Server (`server/`)

## Purpose
Central hub. Receives jobs via REST+WS, queues them in SQLite, spawns AI CLI tools as subprocesses, streams results back to bridges+clients. Manages all state.

## Entry Point & Startup (`src/index.ts`)
1. Load config from env vars
2. Open SQLite DB + run migrations
3. Initialize all 13 repos (jobs, agents, apikeys, users, policies, audit, projects, workers, usage, dependencies, headlessPrograms, settings, sync)
4. Seed defaults on first run (bootstrap admin user with generated/random password saved to `bootstrap-admin.txt` beside DB unless `BOOTSTRAP_ADMIN_PASSWORD` is provided and strong, plus admin API key). Agent configs now start empty by default on first install. Example headless programs can be seeded (`SEED_EXAMPLE_HEADLESS_PROGRAMS=true`, default on). Always seeds coordinator scripts dir (`data/coordinator-scripts/`) and coordinator playbooks dir (`data/coordinator-playbooks/`) with missing defaults on startup (global + blender + godot + houdini + unity + unreal + comfyui starter playbooks).
4.5 Normalize legacy agent args at startup - Codex (`-p`, old approval flags) and Claude Code (`-p`, `--print`, `--model`) - so existing DB configs remain compatible with current CLI behavior.
5. Create WebSocket hub, ProcessTracker, Scheduler, SyncManager
6. Start timeout checker, sync cleanup timer, session cleanup timer (hourly), worker loop, and coordinator training scheduler tick (60s)
7. Create Hono app, mount all routes
8. `Bun.serve()` handles HTTP (→Hono) and WS (at `/ws`) - **Optional TLS** via `TLS_CERT_PATH` + `TLS_KEY_PATH` env vars. **WS auth enforced before upgrade** (API key validated in fetch handler; supports query `key` and `Sec-WebSocket-Protocol: arkestrator.auth.<token>`). WS type/role constraints are enforced (`client` sockets reject `bridge` keys; `bridge` sockets reject `client` keys). Runs with a single listener (no `reusePort`) to avoid split traffic across stale processes. On bridge connect: broadcasts bridge status + upserts worker + **upserts worker_bridges** (program history). Worker identity uses the socket-supplied `workerName` when present, with local fallback derivation (`workerName` → `osUser@ip` → `host-ip` → name/program) only when missing; the server no longer merges/deletes workers solely because sockets share an IP or OS username. On client connect: sends full bridge context sync via `hub.sendContextSync()` + **`hub.sendBridgeStatus()`** + **`hub.sendWorkerStatus()`** (so client gets full state immediately). On bridge disconnect: hub clears context + broadcasts `bridge_context_clear` to clients. On client disconnect: calls `handleClientDisconnect()` from `client-dispatch.ts` to fail any orphaned dispatched local LLM jobs.
9. Graceful shutdown on SIGINT/SIGTERM (clears all timers)

### Job Queue SQL (pickNext)
```sql
SELECT * FROM jobs WHERE status = 'queued'
  AND id NOT IN (
    SELECT d.job_id FROM job_dependencies d
    JOIN jobs j ON j.id = d.depends_on_job_id
    WHERE j.status NOT IN ('completed')
  )
  ORDER BY CASE priority WHEN 'critical' THEN 0 WHEN 'high' THEN 1 WHEN 'normal' THEN 2 WHEN 'low' THEN 3 END, created_at ASC
  LIMIT 1
```

- **process-tracker.ts**: Tracks running subprocesses, 30s timeout check interval, kill on timeout/shutdown. `getProcess(jobId)` exposes the subprocess handle for a given job (used for stdin guidance delivery).
- **file-snapshot.ts**: Recursive directory snapshot (skips .hidden, node_modules, __pycache__) with robust post-run diffing: reconciles fs.watch events with before/after snapshots, reports binary changes as metadata placeholders (`[binary:ext] N.NMB` with file extension detection), and tracks symlink changes.
- **stream-json-parser.ts**: Parses Claude Code `--output-format stream-json --verbose` and Codex CLI `--json` JSONL output. Extracts human-readable display lines for real-time log streaming. Text preview limit: 500 chars. Tool result preview: 500 chars. Formats MCP tool calls (`arkestrator__execute_command`, `arkestrator__create_job`, etc.) with meaningful summaries. Token extraction accumulates per-turn usage from assistant events, extracts from `usage` sub-object in result events, and supports `total_cost`/`cost_usd` aliases for cost reporting. Codex JSONL events handled: `thread.started`, `turn.started`, `item.completed` (accumulates token usage from `usage` fields), `turn.completed`.
- **token-parser.ts**: Regex-based token extraction from stdout (claude-code, codex, gemini patterns)

## Queue System (`src/queue/`)
- **worker.ts**: `WorkerLoop` - adaptive `setTimeout` poll loop (base/min/max cadence). Speeds up when busy or dispatching, backs off when idle. Each tick: check available slots (maxConcurrent - running), pick next job, claim atomically, **check per-user token budget** (if job has `submittedBy` and user has limits, fails job if over budget), load agent config + policies, apply per-job runtime overrides to produce effective spawn config, inject worker's projectRoot if targeting, and fire-and-forget `spawnAgent()`. The tick loop now also collects connected bridge worker names and uses `pickNextForWorker` per worker to ensure worker-targeted jobs only dispatch when the target worker has bridges connected. After the dispatch loop, the worker checks if queued jobs still exist and keeps the poll interval aggressive (`minPollMs`) instead of backing off, preventing dispatch starvation where worker-targeted jobs stay queued for minutes. The loop now also passes the shared worker-resource lease manager into spawned runs so DCC-heavy bridge/headless steps can serialize per worker without globally capping agent concurrency. Tick body wrapped in try/catch for resilience. Periodic diagnostic logging (every 10s) shows slot usage + current poll interval. **`dispatchById(jobId)`** public method allows manual dispatch of a specific queued job, bypassing the poll cycle (used by `POST /api/jobs/:id/dispatch`).
- **scheduler.ts**: Thin wrapper around `jobsRepo.pickNext()` and `jobsRepo.pickNextForWorker(workerName)` for worker-targeted scheduling.

## Workspace Resolution (`src/workspace/`)
- **resolver.ts**: Simplified fallback chain: preferredMode → server default → no projectRoot → explicit projectId prompt injection → local path exists → attached files → command fallback. Projects now only attach for prompt injection (via `formatProjectPrompt()`) rather than workspace resolution; no bridge-path glob matching or project-based workspace mode override. Resolver returns `resolutionStep` + `resolutionReason` for diagnostics; spawner warns when step 5+ fallback branches are used. Jobs that set `editorContext.metadata.project_selection = "none"` skip project lookup + local-path repo resolution and fall through to sync/command branches.
- **command-mode.ts**: Builds command-mode system prompt (tells agent to output fenced code blocks), parses output for code fences, detects bridge type from metadata or `bridgeProgram` fallback (supports godot, blender, houdini, unity, comfyui), derives the expected executable script language, and filters out echoed helper/template snippets (`am`, curl, PowerShell, `# your code here`) before returning commands. ComfyUI command-mode instructions explicitly allow Python-driven filesystem operations for outputs/model installs (while still forbidding Bash/Write/Edit tool usage) and now clarify machine boundaries: destination paths are local to the ComfyUI worker unless artifacts are explicitly transferred to a destination worker bridge.
- **sync-manager.ts**: Creates temp dirs, writes attached files, TTL-based cleanup

## Policy Enforcement (`src/policies/enforcer.ts`)
- `checkPromptFilters()` - regex against prompt text
- `checkCommandScripts()` - regex against command script content (case-insensitive). Filters at 3 choke points: spawner (command-mode output), REST bridge-command endpoint, WS bridge_command_send handler
- `checkEngineModel()` - exact match on engine or engine:model
- `checkFilePaths()` - glob matching via minimatch (normalized to forward slashes)
- `getToolRestrictions()` - returns blocked tool names
- `validateJobSubmission()` - master check combining prompt + engine/model

## Security Controls (`src/security/`)
- `network-policy.ts`: centralized server-side network/rate-limit controls used by routes.
- `worker-rules.ts`: persistent per-machine rule storage/evaluation (`security_worker_rules_v1`) for ban state, per-machine client orchestration allow/deny, IP allow/deny lists, and distributed local-LLM routing controls (`localLlmEnabled`, `localLlmBaseUrl`).

## Sidecar Build
`pnpm build:sidecar` (or `node scripts/build-sidecar.mjs`) compiles the server into a standalone binary via `bun build --compile`. Primary output: `client/src-tauri/binaries/arkestrator-server-{triple}[.exe]`.  The binary embeds the Bun runtime, SQLite, and all dependencies - no Bun or Node.js needed on the target machine. Used by the Tauri client's "Start Local Server" feature in production mode.

## Dev Reset Script
`pnpm --filter server reset:state` (or `bun scripts/reset-state.ts`) performs a local reset for stuck dev sessions:
- kills listeners on ports `7800` (server) and `5173` (client)
- removes `server/data` (SQLite DB + temp state)
- removes shared bridge config `~/.arkestrator/config.json`
- supports `--dry-run` to preview actions
- on Windows, locked files are retried and treated as best-effort (reset continues instead of failing hard)

## MCP Tool Server (`src/mcp/`)
Exposes Arkestrator bridge interaction and job orchestration as MCP tools for external MCP clients (for example Codex app) and coordinator agents. Mounted at `/mcp`.

- **tool-server.ts**: `createMcpServer()` - creates `McpServer` instance with 13 tools. `McpDeps` includes `jobsRepo`, `agentsRepo`, `depsRepo`, optional `processTracker`, `principal` (authenticated user or API key), and request-auth forwarding metadata (`requestOrigin`, `requestAuthHeader`, `requestCookieHeader`) used by client-parity forwarding. Per-tool permission checks: `execute_command`/`execute_multiple_commands`/`run_headless_check` require `executeCommands`, `create_job`/`cancel_job` require `submitJobs`, `submit_job_intervention` requires `interveneJobs`.
  - *Bridge tools:* `execute_command`, `execute_multiple_commands`, `list_bridges`, `list_targets`, `get_bridge_context`
  - *Job orchestration tools:* `list_agent_configs`, `create_job`, `get_job_status`, `get_job_logs`, `list_jobs`, `cancel_job`
  - *Client parity tool:* `client_api_request` (method/path/body) - forwards to an allowlisted set of non-admin client endpoints so MCP can reuse existing route validation/permissions for rich job payloads and training workflows.
  - *Running-job guidance tools:* `submit_job_intervention` and `list_job_interventions` forward to the same intervention APIs used by the client so MCP users can steer queued/paused jobs and supported running jobs with the same delivery-state semantics.
  - `create_job` creates a sub-agent job with optional `target_program`, `target_worker`, `depends_on_job_ids`, `coordination_scripts` (mode + selected script names for per-sub-job coordination script control), **`handover_notes`** (context from parent, prepended to prompt as `## Context from Coordinator`). Sub-jobs default to `coordinationMode: "server"`. Notifies clients via `hub.broadcastToType`.
  - `get_job_status` returns status, file changes, commands, blocking deps, **`output_summary`** (last 20 lines of logs for completed jobs, last 10 for failed).
  - `get_job_logs` returns tail logs for any job with bounded line limits (`lines` arg, max 2000), plus `total_lines`/`returned_lines`.
  - `list_jobs` returns recent jobs (up to 50) with optional status filter.
  - `cancel_job` cancels queued/paused/running jobs; for running jobs it kills tracked subprocesses first when process tracking is wired.
  - `list_targets` returns orchestration-ready targets as a union of currently connected bridges and enabled headless program configs.
  - `run_headless_check` now uses shared `runHeadlessCheck()` from `routes/bridge-commands.ts` so MCP and REST/CLI return consistent behavior/output.
  - All tools call shared helpers from `routes/bridge-commands.ts` or repos directly.

- **routes.ts**: `createMcpRoutes()` - Hono route handler at `/mcp`. Uses custom `StatelessTransport` (in-memory, Bun-compatible - no Node.js `http.ServerResponse` dependency). Each POST creates a fresh transport, connects to MCP server, delivers JSON-RPC message, captures response. Auth uses `principalHasPermission("useMcp")` uniformly for both user sessions and API keys (was role-based for API keys); auto-provisioned user API keys (`auto:user:<id>`) are also gated by that same user permission. Passes `principal` into `McpDeps` for per-tool permission enforcement. **Bug fix**: MCP notifications (no `id` field, e.g. `notifications/initialized`) now return HTTP 202 immediately instead of hanging - `handleMessage()` checks for presence of `id` field and returns `null` for notifications.

### How agents discover MCP tools
The spawner writes `.mcp.json` to the job's CWD before spawning Claude Code agents. Claude Code auto-discovers MCP servers from this file. The config is merged with any existing `.mcp.json` and restored/removed after the job completes.
