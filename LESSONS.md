# Lessons Learned

Patterns and mistakes captured during development. Agents should review this at session start and update it after any user correction.

## Patterns

- **For containerized CLI auth issues, prioritize in-app runtime-context controls over host-shell instructions**: when users run Arkestrator in Docker/TrueNAS, login guidance must target the exact runtime user/context jobs execute under (or provide one-click admin controls) instead of generic host-side CLI instructions.
- **Treat repo publication actions as explicitly gated**: Never run private/public push, mirror, or sync git commands unless the user asks in that turn, and always request a final confirmation immediately before execution.
- **Do not assume bootstrap credentials remain current**: `bootstrap-admin.txt` can be stale after password rotations; for automated admin UI debugging, use confirmed current credentials (or create a fresh session token) before diagnosing page behavior.
- **Fix interaction bugs without changing expected UX modality**: If a user reports broken modal controls, preserve popup behavior and harden interaction handling (selection/close events) instead of replacing modal workflows with inline panels unless explicitly requested.
- **Match action labels to outcomes**: Controls named `Use Template` should perform an immediate add/create action unless the UI explicitly says it opens a draft/customization step.
- **Harden UI edit flows against legacy persisted shapes**: Admin editors should normalize API payloads (for example `args` as string vs array) before binding forms so old DB rows can’t silently break click handlers.
- **For critical admin actions, prefer non-modal fallbacks when modal state is flaky**: If repeated reports indicate click-trap behavior and no clear runtime errors, move create/edit flows to inline panels to guarantee operability while preserving modal UX only where required.
- **Embedded webview fallback for confirmations**: In Tauri/embedded surfaces, if custom confirm overlays are unreliable, use `window.confirm` as a deterministic fallback for destructive actions while root interaction issues are stabilized.
- **Prefer inline pickers over modal pickers in embedded webviews when interactions regress**: Template/selection flows in embedded admin contexts should default to inline panels after repeated modal lockups.
- **Split-pane UX changes need target-page verification**: After implementing rail/detail layout updates, always validate the specific tab/page the user reported (not a related settings page) and confirm right-pane content appears under the expected interaction (for example `Edit` click opens editor).
- **Interpret “right panel” literally when layout already has a right column**: If the user asks for content “on the right of what we have now,” implement an additional side pane (main content + editor pane) rather than replacing existing right-column content.
- **Avoid idle placeholder panes for optional tools**: If an edit pane is not needed until a user action, keep it unmounted by default and render it only after explicit intent (for example clicking `Edit`).
- **Coordinator script layout clarity**: Keep only the global coordinator script above bridge selection; place bridge-specific script editor below the selected bridge context so users immediately see which bridge they are editing.
- **Analyze flow visibility and usefulness**: Source analysis actions should be queued/trackable as explicit jobs in the coordinator UI, generate meaningful per-project artifacts (machine JSON + human-readable notes), and keep dense source/job lists collapsed by default with foldouts.
- **Coordinator notes quality bar**: Generated project notes must include concrete project contents (inventory/type breakdown/key files/sample paths), not a shallow template. If users ask for “analyze,” output must be materially useful as future agent guidance.
- **Fast vs AI separation**: If users ask for true AI analysis, expose explicit mode selection and route AI mode through real queued agent jobs (bridge-backed when program-specific), not an immediate deterministic helper.
- **Toggle/checkbox UI integration**: Form labels that contain checkboxes/radios must explicitly force row layout (`flex-direction: row`) and aligned spacing; relying on generic label styles causes stacked/clunky controls. Keep checkbox/radio styling consistent at app-global CSS level across client and admin.
- **Do not apply full-width form rules to checkboxes/radios**: Page-level selectors like `input { width: 100% }` will override global control styling and break toggle/radio layout. Scope full-width rules to text-like inputs only (`input:not([type="checkbox"]):not([type="radio"])`).
- **Synthetic jobs must satisfy protocol schemas**: Any internally queued job (analyze/training/scheduler) still flows through WS `Job` validation. If `editorContext` is present it must include required fields (`projectRoot`), otherwise clients silently drop `job_updated`/`job_list_response`.
- **Training actions must produce visible artifacts**: If a job is named “training,” it should update an inspectable artifact (not only transient in-memory/script output) and emit explicit changed/no-change log lines so users can verify outcomes quickly.
- **Context event handling must be idempotent**: Bridge context events can be replayed or duplicated (reconnects/menu wiring). Store and relay context items as upserts keyed by per-bridge `@index`, not append-only, to avoid duplicate rows like repeated `@1`.
- **Do not depend on optional bridge metadata for worker persistence**: Worker history/offline targeting must survive missing `workerName`. Derive a stable fallback identity on server connect and always refresh worker status on connect/disconnect.
- **Bridge command attribution must be job-aware outside MCP**: MCP `execute_command` updates `usedBridges`, but CLI/REST bridge commands need explicit job correlation (`X-Job-Id`) too, otherwise multi-bridge jobs appear as single-bridge in Jobs UI.
- **Do not reconnect bridges just to refresh file/path metadata**: Reconnect-on-file-open introduces thread/socket race risk and can make bridges disappear from the UI. Prefer updating bridge name/project metadata from regular editor-context events while keeping the socket stable.
- **Bridge lists must preserve offline history**: If UI bridge lists are based only on live sockets, closing Blender/Houdini looks like the bridge vanished. Always merge live bridge status with persisted worker program history (`knownPrograms`) so offline targets remain visible/selectable.
- **Do not let `worker_status` overwrite bridge connectivity state**: Worker snapshots and bridge snapshots can arrive in different order. Treat bridge connectivity as authoritative from bridge-specific status payloads/REST bridge lists, otherwise online bridges can be incorrectly shown as offline.
- **Stabilize UX scope before deleting legacy UI files**: If a surface is being slimmed down, keep dormant pages/components until the replacement state (controls/workflows) is validated with the user, and track hard deletion as an explicit TODO item.
- **Local server status must be health-driven, not spawn-driven**: Client server status cannot assume it only manages its own child process. Always probe localhost health in the background so manually started dev servers are reflected as running.
- **Keep agent instruction docs aligned with current product scope**: When app responsibilities change (for example admin no longer mirroring client job flows), remove outdated directives from both `AGENTS.md` and `CLAUDE.md` in the same pass to avoid future regressions.
- **Do not assume users want fallback behavior while debugging local-model failures**: If a user reports a bad local run, first explain the exact local execution/context gap from logs and metadata; only discuss fallback paths as optional follow-up, not as implied intent.
- **Do not hardcode task-specific gates when users ask for learning behavior**: If repeated failures exist (for example missing a domain-specific node), prioritize fixing generic learning/indexing/retrieval paths so the system adapts from artifacts/outcomes, rather than adding one-off hardcoded workflow rules.
- **Provider model lists are temporally unstable**: UI model/reasoning suggestion lists for Codex, Claude, Gemini, local providers, etc. must be treated as perishable data. If they are not dynamically discovered, re-verify them against current official provider/runtime sources whenever touching that UI and before release/build packaging work; do not assume previously hardcoded entries are still current.
- **If the runtime owns provider metadata, do not re-hardcode it in Arkestrator**: For model pickers/defaults, prefer Claude/Codex app or CLI state (cache, local runtime artifacts, provider priority metadata) over server-maintained slug lists, and use that runtime-owned signal to choose the default/latest model.
- **Command-mode success requires executable output, not just exit code 0**: Treat jobs that produce zero fenced commands in command mode as failures with actionable guidance; otherwise users get misleading "completed" states with no result.
- **When users ask for “all training files,” prioritize one unified explorer surface**: expose a single global vault/tree with clear subfolders (for example `scripts/`, `playbooks/`, `learning/`) instead of scattering controls across separate pages/endpoints.
- **Training exchange actions must be file-download/file-upload flows, not vault-write actions**: when users ask to export/import training data between servers, default to portable zip bundles (download + file picker ingest) and keep in-vault save/edit controls separate.
- **Generated training artifact metadata must carry actor identity**: for server-generated `learning/jobs/...` artifacts, persist at least submitter id and machine context when available so Admin metadata never degrades to `Created By unknown` for attributable jobs.
- **Houdini menu coverage must be validated per surface**: `OPmenu.xml` hooks node/network RMB only; viewport selection and parm workflows need their own menu hooks (`SelectCustomMenu`, `PARMmenu`, or global main-menu action) to truly support “from anywhere”.
- **Keep Houdini context-action labels and viewport hooks aligned**: If users expect the exact `Add to Agent Context` action in viewport RMB flows, use the same label across all surfaces and include pane/viewport tab menu hooks in addition to node/parm/component menus.
- **Houdini select-state RMB is a separate surface**: The `SELECT (to overlay)` viewport popup is not controlled by OP/PARM/PaneTabType XML hooks; ensure it is explicitly handled (for example via a runtime Qt menu hook) when users ask for true “any context” right-click coverage.
- **Houdini packages must be relocatable across OSes**: Do not hardcode bridge package roots to `HOUDINI_USER_PREF_DIR` or a single-platform fallback path. Use Houdini package-relative discovery (`$HOUDINI_PACKAGE_PATH`, `hpath`) and cross-platform user-pref fallback logic so the same shipped package layout works on Windows, macOS, and Linux.
- **Tauri icon generation requires square sources**: When regenerating desktop app icons from a brand SVG, always normalize to a square-centered source first; `tauri icon` rejects non-square assets and can silently leave old icon outputs in packages if not regenerated from a square file.
- **Keep runtime brand assets in sync with docs masters**: Updating installer icons is not enough; the startup/setup/sidebar UI imports `client/src/assets/brand/*`, so when the logo source changes, sync those client brand SVGs from `docs/assets` as part of the same task.
- **Bridge auth rotation must be runtime-resilient**: Reset/reinstall steps are not enough when users switch servers or rotate keys; bridge WS clients should re-read `~/.arkestrator/config.json` during reconnect/read polling and trigger reconnect when shared `apiKey`/followed `wsUrl` changes.
- **Client owns canonical worker identity**: When the desktop client is the source of machine identity, bridges should consume `workerName` from shared client config rather than inventing their own hostname fallback. Otherwise worker records drift between client and bridge and the wrong machine label wins in Workers.
- **Machine linking must use persistent IDs, not IPs or mutable hostnames**: When joining desktop clients and bridges into one worker, use a client-owned persistent `machineId` propagated through shared config and WS metadata. IPs and hostnames are too unstable to be the canonical join key.
- **Do not fix shared machine identity in only one bridge**: Once worker identity is client-owned, every maintained bridge must follow the same `workerName`/`machineId` shared-config logic. Leaving one plugin on hostname fallback recreates duplicate-worker bugs immediately.
- **DCC headless execution must stay client-side, never server-side**: If a user asks for Blender/Houdini/Godot CLI or headless execution, the server must not launch those app binaries from its own PATH or filesystem context. The server should route the request to the target client/worker, and if that worker cannot perform headless execution it must fail clearly instead of silently falling back to server-local execution or a live GUI session.
- **Coordinator knowledge must be server-owned and portable**: Do not store durable playbook/reference links as raw machine-local absolute paths when the server needs them later. Import bounded portable artifacts into server-managed storage and keep cleanup scoped/manual.
- **Container tag semantics must match workflow triggers**: If deployments are expected to follow `:main`, the image publish workflow must actually run on `main` pushes and emit a `main` tag. Otherwise `:latest` and `:main` silently diverge and remote debugging becomes misleading.
- **Pin GitHub Actions to real published tags when CI is already brittle**: If a floating major alias like `pnpm/action-setup@v4` fails to resolve in production CI, switch to the current concrete release tag (for example `v4.2.0`) instead of assuming the alias exists everywhere.
- **Do not label machine OS accounts as signed-in app users**: Worker/bridge metadata (`osUser`) represents the remote machine account, not the authenticated Arkestrator client account; UI labels must distinguish both explicitly.
- **For job-ID debugging, resolve server origin before querying local state**: If a user reports a job from a remote deployment (for example TrueNAS), pull the job via that server API first; local Tauri DB lookups can be empty/misleading.
- **When remote/server context is explicit, avoid mixed-database debugging flow**: Do not pivot into local desktop DB inspection before confirming the remote API path and job state from the active deployment (for example `truenas.local:7800`), even if local DB access is convenient.
- **When a user says remote SSH is available, verify it before claiming otherwise**: On appliance-style hosts like TrueNAS, test the host-specific key and likely usernames (for example `root`) first instead of assuming SSH is unavailable from the current machine.
- **Treat concrete runtime stderr as authoritative over secondary assumptions**: if a user provides direct CLI evidence like `--dangerously-skip-permissions cannot be used with root/sudo privileges`, prefer that observed runtime behavior immediately and correct the integration/docs before leaning on recollection or generalized provider docs.
- **Verify the deployed runtime user, not just the compose snippet in chat**: when debugging Docker/TrueNAS permission behavior, inspect the live container (`docker inspect ... Config.User`, `docker exec ... id`) before assuming a YAML edit actually changed how the running service launches.
- **Container runtime user changes can hide Bun-global CLIs**: if a Docker image installs `claude`/`codex` with `bun install -g` as root and later runs as a non-root user, those binaries may disappear from runtime `PATH`. Install Bun globals into a system-wide `BUN_INSTALL` location before switching users.
- **Job detail prompts need explicit readability affordances**: Prompt blocks should include clear contrast plus copy/expand controls so users can inspect the exact submitted prompt without relying on raw scrolling.
- **Jobs filter debugging must verify both payload labels and fallback inventories**: If users report empty/opaque Jobs dropdowns, check both the job payload fields (`submittedByUsername`, `workerName`, `targetWorkerName`) and whether the UI should fall back to live worker inventory instead of assuming current rows contain enough display data.
- **Running-job guidance must reach the active agent, not just the database/UI**: When users say “guide a running job,” the implementation needs a real delivery path into the current execution loop (for example agent-side polling or server-managed turn injection) and delivery state must flip only when the running job actually consumes the note.
- **Do not use self-referential effects for UI accordion state unless re-open behavior is explicitly intended**: if a `$effect` both reads and repopulates expand/collapse state, user clicks can be immediately undone on the next reactive pass. For simple foldouts, prefer the direct `expanded === key ? null : key` toggle pattern used by the working accordion.

- **Token tracking must cover ALL completion paths**: When shared cleanup logic (e.g. token recording, status updates) lives at the end of a long function with many early returns, extract it into an idempotent helper function and call from every exit path. A flag guard (`tokensRecorded`) prevents double-writes. Simply placing cleanup code at the bottom of a function is insufficient when 11+ early returns bypass it.
- **Unicode smart quotes break worker name matching**: macOS hostnames use Unicode RIGHT SINGLE QUOTATION MARK (U+2019) which does not match ASCII apostrophe (U+0027). Always normalize smart quotes to ASCII in identity/comparison code -- apply normalization at slug generation, DB storage, and runtime set-membership checks to ensure consistent matching regardless of input source.
- **CORS defaults must be restrictive, not permissive**: When `CORS_ORIGINS` is unset, default to localhost/Tauri dev origins only. A fully open CORS policy (wildcard or empty-means-allow-all) is a production security risk even if the app is behind a reverse proxy.
- **API key prefix changes need backwards compatibility**: When changing key prefixes (e.g. `am_` to `ark_`), always accept both old and new prefixes in validation so existing keys continue to work. Only generate new keys with the new prefix.
- **pnpm-workspace.yaml corruption from bad writes**: Always verify `onlyBuiltDependencies` entries are full package names, not single characters - a corrupted write can turn `"esbuild"` into `["e","s","b","u","i","l","d"]`.
- **Standardize error responses before public launch**: Inconsistent error shapes (some routes returning `{ error: string }` while others use `errorResponse()`) break SDK consumers. Audit all route files for consistent structured error codes before shipping.
- **Environment variable documentation must be comprehensive**: `.env.example` files are often the first thing deployers read. Every supported env var should be documented with defaults and descriptions, not just a subset.
