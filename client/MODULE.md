# Module: Client (`client/`)

## Purpose
Primary user-facing desktop dashboard (Tauri v2 + Svelte 5). Users manage jobs, configure agents, view workers, manage projects. Connects to server via REST + WebSocket.

## App Structure
- `App.svelte` - Root: TitleBar + (Legal Gate OR Boot Screen OR BootstrapWizard OR Setup OR Sidebar+Content+StatusBar) + Toast overlay. **First-run legal gate** blocks startup until user accepts pre-release disclaimer/no-warranty terms (persisted locally with versioned key). **Boot sequence**: checks saved `serverMode` - if `"local"`, auto-starts server and waits for "Server listening" in stdout before validating session; if `"remote"`, validates session immediately via `api.auth.me()`. Session validation restores `allowClientCoordination`, `clientCoordinationEnabled`, and `canEditCoordinator` flags and triggers local capability probing when allowed. Shows boot screen with progress message during startup. **Routing priority**: if wizard not complete and no session → BootstrapWizard; if wizard complete but no session → Setup (returning user reconnection); if authenticated + pendingWizard → BootstrapWizard.
- Routing: simple `nav.current` string matching (`chat`, `jobs`, `admin`, `workers`, `projects`, `coordinator`, `settings`). Default page: `chat`.
- Tauri v2 native window with custom titlebar, frameless
- Global theme CSS (`client/src/app.css`) includes unified custom checkbox/radio control styling (shape, checked indicator, focus ring, disabled state) for consistent non-clunky toggles across pages.

- **rest.ts**: `api` object with namespaced methods (jobs.list, jobs.create, jobs.cancel, `jobs.setOutcome` (accepts `good/average/poor` and legacy `positive/negative`), `jobs.interventions(id)`, `jobs.intervene(id, payload)`, `jobs.clearPendingInterventions(id)`, `jobs.listArchived()`, `jobs.listTrashed()`, `jobs.archive(id)`, `jobs.restore(id)`, `jobs.permanentDelete(id)`, agents.create/update/delete, `agents.localModels(runtime, targetWorkerName?, host?)` returning local model catalog metadata (`allowedModels`, `catalog`); Uses `fetch()` with Bearer token from connection store. Request helper now skips forced `Content-Type: application/json` for `FormData` bodies (required for multipart uploads), and includes an SSE POST helper used for streamed local-model pull progress. Coordinator upload supports optional per-file relative `paths` so folder uploads preserve nested structure on server. `api.apiBridges` namespace: `list()`, `presets()`, `get(id)`, `create(data)`, `update(id, data)`, `delete(id)`, `test(id)`, `actions(id)` for managing external REST API bridge configurations. `api.chat.stream()` is the SSE client for `/api/chat`, and now supports optional `runtimeOptions` and optional `jobIds` (string array, up to 20 recent job IDs for context) in the request body.
- **ws.ts**: WebSocket manager. `connect(url, apiKey)` opens WS at `url/ws?type=client` and sends API key via subprotocol `arkestrator.auth.<apiKey>`. Auto-reconnect with exponential backoff (3sâ†’30s, **with 0-25% random jitter** to prevent thundering herd). **Stale-socket guard**: each connection has a generation ID; `onopen/onmessage/onclose/onerror` ignore events from superseded sockets, preventing reconnect loops where an old socket close schedules reconnect after a newer socket already connected. Reconnect scheduling also skips while a socket is already `OPEN`/`CONNECTING`. **Status-message resilience**: if strict `Message` parsing fails but payload is recognizable `worker_status`/`bridge_status`, client applies a fallback status dispatch, rebuilds offline bridge history, and schedules a short REST reconcile (`/api/workers`) so new worker/bridge connections still appear live without restart. `dispatch()` routes incoming messages to stores. Bridge status handling synthesizes persistent offline bridge entries from worker `knownPrograms`, so closed bridges remain selectable/listed as offline instead of disappearing. Worker updates now rebuild offline history from the current online bridge snapshot and then reconcile via REST to avoid missed/racy status events. The WS layer also now accepts `worker_headless_command`, invokes the native `run_worker_headless` Tauri command locally, and sends `worker_headless_result` back to the server so worker-scoped Blender/Houdini/Godot CLI runs happen on the client machine rather than the server. The WS layer also dispatches `client_job_dispatch`, `client_tool_result`, and `client_job_cancel` messages to `clientJobManager` for client-dispatched local LLM job execution, and connects/disconnects the job manager on WS lifecycle events. Exposes `isLocalWorker()` so UI can reliably identify the row that corresponds to the current desktop machine using the same `machineId`/name identity sent over WebSocket. `job_intervention_updated` now updates the jobs intervention cache and mirrors operator-note events into chat (without `jobId` on sub-job status and intervention chat messages to prevent duplicate job cards). On connect: requests job_list + agent_config_list + writes `~/.arkestrator/config.json` locally via `writeLocalConfig()` using Tauri IPC command (`invoke("write_shared_config")`) for bridge auto-discovery on same machine, including the client-owned `workerName` and persistent `machineId`. When the selected server is remote, that Tauri write now also starts/updates a client-owned localhost relay and writes bridge-facing localhost URLs while preserving the real remote server URLs in `remoteServerUrl` / `remoteWsUrl`. This replaces the need for direct same-machine bridge connections to the remote host and fixes remote-server + local-bridge cases where specific DCC GUI apps cannot route to the remote endpoint directly. **P2P file transfer**: `handleTransferServeRequest()` handles `transfer_serve_request` by invoking `fs_serve_files` Tauri command to start a local HTTP file server, then calls `detectLanIp()` to discover the machine's LAN address, and replies with `transfer_serve_ready`. `handleTransferInitiate()` updated to attempt P2P download from `p2pUrl` first when present, with automatic fallback to server URL on failure. `detectLanIp()` resolves the client's LAN-reachable IP address for P2P connectivity. **Validates saved session** via `api.auth.me()` on connect - clears session on failure, refreshes coordination flags, and triggers capability probing when allowed.

| `client_job_cancel` | Cancel a client-dispatched job via `clientJobManager.handleCancel()` |
| `file_deliver` | Write delivered files to disk via `fs_apply_file_changes` Tauri command |
| `transfer_initiate` | Handle HTTP file transfer request from server — supports P2P mode (downloads from `p2pUrl` with fallback to server URL on failure) and standard mode (`fs_download_file` or `fs_upload_file` Tauri commands). Sends `transfer_progress` back to server |
| `transfer_serve_request` | Handle P2P serve request from server — invokes `fs_serve_files` Tauri command to start a local HTTP file server (port range 17830-17850), detects LAN IP via `detectLanIp()`, and replies with `transfer_serve_ready` (host, port, tokens) or error |
| `task_progress` | Update task job progress (percent, statusText) via `jobs.upsert()` |
| `error` | Set `connection.lastError` |

## Tauri Config (`src-tauri/`)
- `tauri.conf.json`: frameless window, custom titlebar, size 1200x800, bundle identifier `com.arkestrator.desktop`, and explicit updater plugin config (`plugins.updater.pubkey`, `plugins.updater.endpoints`) so updater plugin initialization is stable in dev/release config merges.
- `../scripts/ensure-sidecar.mjs`: validates expected sidecar binaries for the current host triple and triggers `pnpm --filter @arkestrator/server build:sidecar` if missing. It also ensures admin UI resources are bundled by copying `admin/dist` into `client/resources/admin-dist` before Tauri packaging/dev launch.
- `src/main.rs`: standard Tauri entry point
- `src/bridges.rs`: Bridge plugin management commands — `fetch_bridge_registry` fetches registry.json from GitHub Releases with 5-min TTL caching, `download_and_install_bridge` downloads and extracts zip to target path, `detect_program_paths` auto-detects DCC installations via glob hints, `get_installed_bridges`/`save_bridge_installation`/`uninstall_bridge` manage `~/.arkestrator/bridges.json` tracking, `check_path_exists` validates paths. Cross-platform path expansion for `~`, `%APPDATA%` etc. `detect_headless_programs` auto-detects installed DCC headless executables (Blender, Houdini hython, Godot) using `detect_executable_paths()` helper with cross-platform search patterns and `extract_version_from_path()` for version inference. Returns `DetectedHeadlessProgram` structs (program, path, version).
- `src/comfyui.rs`: ComfyUI process management commands — `detect_comfyui_paths` (cross-platform installation detection), `launch_comfyui` (launch as background process with managed PID), `stop_comfyui` (stop managed process), `is_comfyui_running` (check if managed process is alive), `check_comfyui_nodes` (check if arkestrator custom nodes are installed), `get_comfyui_autostart`/`set_comfyui_autostart` (read/write auto-start preference to config.json).
- `src/lib.rs`: custom Tauri commands - `write_shared_config` writes `~/.arkestrator/config.json` from the client's local machine, includes the canonical `workerName` plus persistent `machineId` for same-machine bridge reuse, auto-starts/restores a localhost bridge relay for remote servers so same-machine DCC bridges can target `127.0.0.1:<relay-port>` instead of the remote host directly, and now refuses malformed `apiKey` overwrites so a bad desktop-side value cannot clobber bridge auth during reconnect. The shared config preserves `remoteServerUrl` / `remoteWsUrl` for diagnostics and relay restore on next app launch. `get_machine_identity` prefers macOS `ComputerName`/`LocalHostName` via `scutil` before generic hostname fallback so worker registration uses the machine label users expect and generates a UUID-backed `machineId` persisted in shared config; `get_local_hardware_capability` returns native CPU/RAM/GPU hints for client-coordination capability probing; `list_local_ollama_models` probes desktop-local Ollama catalogs; `pull_local_ollama_model` performs desktop-local model download/update via Ollama CLI; `run_worker_headless` executes worker-local headless runs with timeout enforcement and script placeholder expansion so the desktop client can own Blender/Houdini/Godot CLI execution on behalf of the server. Filesystem commands: `fs_apply_file_changes` (batch apply file changes from delivery), `fs_create_directory`, `fs_write_file`, `fs_read_file_base64`, `fs_delete_path`, `fs_exists` — used by the `file_deliver` WS handler for cross-machine file delivery. HTTP transfer commands: `fs_download_file` (downloads a file from a server URL to a local path), `fs_upload_file` (uploads a local file to a server URL) — used by the `transfer_initiate` WS handler for HTTP-based file transfers. P2P transfer commands: `fs_serve_files` (starts a local HTTP file server on port range 17830-17850 serving specified files with token-based auth; returns the bound port) — used by the `transfer_serve_request` WS handler for peer-to-peer file transfers. Depends on `base64 = "0.22"` crate. Registers plugins for shell, autostart, and updater; also sets up tray icon behavior (Show/Hide/Open Dashboard/Stop Server/Quit menu + close-to-tray window handling). "Open Dashboard" opens the admin page in the default browser (reads server URL from shared config, cross-platform via `open`/`cmd start`/`xdg-open`). "Stop Server" sends a POST to `/api/server/shutdown` on a background thread. ComfyUI integration: calls `comfyui::auto_start_comfyui_if_configured()` in `.setup()` hook and `comfyui::shutdown_comfyui()` on tray quit.
- `vite.config.ts`: injects `__CLIENT_VERSION__` plus `__CLIENT_BUILD__` (`<package version>+<git short sha>`) for visible in-app build display.

## Key UX Patterns
- Job list: compact rows with status dot, program icon (G/B/H), name or truncated prompt, worker name, short ID, relative time. Task-mode jobs show a `#T<N>` task badge in the row.
- Job detail panel: task-mode jobs display a progress bar (0-100%), task status text, and task ref (`#T<N>`) in the detail view
- Job tree: dependency-based nesting (children indented under parents)
- Resizable list panel (250px min, drag handle)
- Multi-select with checkboxes, bulk delete
- Status filter tabs (all/paused/queued/running/completed/failed/cancelled)
- "Start Queue" button resumes all paused jobs
