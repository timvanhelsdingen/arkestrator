# Module: Client (`client/`)

## Purpose
Primary user-facing desktop dashboard (Tauri v2 + Svelte 5). Users manage jobs, configure agents, view workers, manage projects. Connects to server via REST + WebSocket.

## Recent Updates (2026-03-19)
- Admin 404 fix on installed clients (2026-03-19): `src-tauri/src/lib.rs` `resolve_admin_dist_path` now checks `_up_/resources/admin-dist` in addition to existing candidates. Tauri v2 rewrites `../` resource paths to `_up_/` in the bundle (all platforms), so the bundled admin SPA was never found and `/admin` returned 404.

## Recent Updates (2026-03-18)
- TOTP 2FA support (2026-03-18): Added two-factor authentication across client. `lib/api/rest.ts` adds three TOTP endpoints: `totpSetup()`, `totpVerifySetup(code)`, `totpDisable(password, code?)`. `lib/stores/connection.svelte.ts` now tracks `totpEnabled` in session state. New deps: `qrcode`, `@types/qrcode`.

## Recent Updates (2026-03-16)
- Client-dispatched local LLM job support (2026-03-16): new `lib/services/` directory with three modules. `ollamaClient.ts` is a thin HTTP wrapper for the Ollama API (`ollamaGenerate()`, `ollamaHealthCheck()`). `localAgenticLoop.ts` implements the client-side agentic loop (`runClientAgenticLoop()`) that calls local Ollama, parses protocol responses via shared `parseLocalAgenticAction()`, and proxies tool calls to the server via WS using shared `runAgenticLoop()` from `@arkestrator/protocol` with client-specific DI deps. `clientJobManager.ts` manages client-dispatched jobs: handles `client_job_dispatch`/`client_tool_result`/`client_job_cancel` WS messages and tracks pending tool requests. `lib/api/ws.ts` now dispatches `client_job_dispatch`, `client_tool_result`, and `client_job_cancel` messages and connects `clientJobManager` on connect/disconnect.
- Chat message persistence fix (2026-03-16): `lib/stores/chat.svelte.ts` `addMessage()` now calls `persistNow()` instead of the debounced `persist()`, ensuring new messages are saved to localStorage immediately rather than risking loss during rapid SSE streaming.
- External server stop support (2026-03-16): `lib/stores/server.svelte.ts` `canStop` now returns `true` for externally-detected servers (not just client-spawned ones). `stop()` calls the server's `POST /api/server/shutdown` endpoint for external servers, enabling the UI kill-server button to work regardless of how the server was started.

## Tauri Config (`src-tauri/`)
- `tauri.conf.json`: frameless window, custom titlebar, size 1200x800, bundle identifier `com.arkestrator.desktop`, and explicit updater plugin config (`plugins.updater.pubkey`, `plugins.updater.endpoints`) so updater plugin initialization is stable in dev/release config merges.
- `../scripts/ensure-sidecar.mjs`: validates expected sidecar binaries for the current host triple and triggers `pnpm --filter @arkestrator/server build:sidecar` if missing. It also ensures admin UI resources are bundled by copying `admin/dist` into `client/resources/admin-dist` before Tauri packaging/dev launch.
- `src/main.rs`: standard Tauri entry point
- `src/bridges.rs`: Bridge plugin management commands — `fetch_bridge_registry` fetches registry.json from GitHub Releases with 5-min TTL caching, `download_and_install_bridge` downloads and extracts zip to target path, `detect_program_paths` auto-detects DCC installations via glob hints, `get_installed_bridges`/`save_bridge_installation`/`uninstall_bridge` manage `~/.arkestrator/bridges.json` tracking, `check_path_exists` validates paths. Cross-platform path expansion for `~`, `%APPDATA%` etc.
- `src/lib.rs`: custom Tauri commands - `write_shared_config` writes `~/.arkestrator/config.json` from the client's local machine, includes the canonical `workerName` plus persistent `machineId` for same-machine bridge reuse, auto-starts/restores a localhost bridge relay for remote servers so same-machine DCC bridges can target `127.0.0.1:<relay-port>` instead of the remote host directly, and now refuses malformed `apiKey` overwrites so a bad desktop-side value cannot clobber bridge auth during reconnect. The shared config preserves `remoteServerUrl` / `remoteWsUrl` for diagnostics and relay restore on next app launch. `get_machine_identity` prefers macOS `ComputerName`/`LocalHostName` via `scutil` before generic hostname fallback so worker registration uses the machine label users expect and generates a UUID-backed `machineId` persisted in shared config; `get_local_hardware_capability` returns native CPU/RAM/GPU hints for client-coordination capability probing; `list_local_ollama_models` probes desktop-local Ollama catalogs; `pull_local_ollama_model` performs desktop-local model download/update via Ollama CLI; `run_worker_headless` executes worker-local headless runs with timeout enforcement and script placeholder expansion so the desktop client can own Blender/Houdini/Godot CLI execution on behalf of the server. Registers plugins for shell, autostart, and updater; also sets up tray icon behavior (show/hide/quit menu + close-to-tray window handling).
- `vite.config.ts`: injects `__CLIENT_VERSION__` plus `__CLIENT_BUILD__` (`<package version>+<git short sha>`) for visible in-app build display.
